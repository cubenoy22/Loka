# 現状設計の弱み・懸念事項リスト（2025-05-03 時点）

1. **UI の動的再構築・差分更新が未対応**

   - App::rerender は setScene 時のみ呼ばれ、状態変化による UI 再構築・再レンダリングは行わない
   - 動的な UI 切り替えや部分的な再描画が必要な場合は、Scene の中で Scene を切り替える等の工夫が必要

2. **複雑な UI ツリー・ネスト構造の拡張性**

   - Scene の中で Scene を持つことで if 分岐や動的 UI は実現できるが、複雑なネストや再利用性の高い UI パターンは今後の設計課題

3. **イベント伝播・コールバックの仕組み**

   - 現状は onClick 等の関数ポインタのみ。より複雑なイベント伝播や親子間通信は今後の拡張テーマ

4. **レイアウト・スタイルの柔軟性**

   - レイアウトやスタイル指定は未実装。今後 Grid/Row/Column 的な部品やスタイル指定 API の追加が必要

5. **テスト性・デバッグ性**

   - UI 状態のダンプやスナップショットテスト、UI ツリーの可視化などは未対応

6. **例外安全性・エラー処理**

   - new 失敗時や不正な Prop 参照など、エラー処理・例外安全性は今後強化の余地あり

7. **C++98 の制約**

   - スマートポインタやラムダ式が使えないため、手動 delete や関数ポインタ設計に注意が必要
   - テンプレートや RTTI の限界も意識

8. **Renderer の仮実装**
   - 実際の描画処理はまだ仮実装。プラットフォームごとの実装や高機能化は今後の課題

9. **再描画最適化・バッチング・差分検知**（優先度：低・将来的な拡張テーマ）

   - State の変更が多い場合、伝播・再描画のバッチングや差分検知による最小限の再描画が必要
   - begin/end 方式は複雑化するため現状は見送りだが、将来的な拡張余地あり
   - ※現状の設計（Page が State を RAII で保持）では、伝播・再描画の複雑化リスクは低く、優先度は下がる

10. **副作用のタイミング制御・多重発火防止**（優先度：低・将来的な拡張テーマ）

    - State/DerivedProp の変更に応じた副作用（API 呼び出し等）の発火タイミング管理が課題
    - commit 境界での副作用実行や多重発火防止の仕組みが必要
    - ※現状の設計では副作用の発火タイミングは Page 内で閉じており、複雑な伝播は発生しにくい

11. **循環参照の静的検知・実行時ガード**（優先度：低・将来的な拡張テーマ）

    - DerivedProp/State の依存グラフに循環が発生しないよう、静的検知や最大伝播深度による実行時ガードを検討
    - ※現状の設計では依存関係が Page 内で閉じているため、循環リスクは限定的

12. **State リスナーのクリーンアップ・メモリリーク対策**（優先度：低・現状はリスク低）

    - Scene 破棄やスレッド跨ぎ時にリスナーが残らないよう、明示的なクリーンアップ処理が必要
    - ※Page が State を RAII で保持し、外部から直接提供しない設計のため、リスナー残存・メモリリークのリスクは大幅に低減

13. **非同期副作用のキャンセル制御**（優先度：低・将来的な拡張テーマ）

    - 非同期処理（API, タイマー等）で古い State に基づく副作用が遅れて発火しないよう、キャンセル制御の仕組みを検討
    - ※現状の設計では副作用の発火範囲が限定的なため、優先度は低い

14. **依存グラフの可視化・伝播ログ**（優先度：低・将来的な拡張テーマ）
    - State/DerivedProp の依存関係や伝播経路が複雑化した場合のため、依存グラフの可視化や伝播ログ出力の仕組みがあると開発効率が向上
    - ※現状の設計では依存関係がシンプルなため、優先度は低い
