# DSLノードの所有権とクリーンアップ

C++98でDSLを構築している関係で、`operator<<` を多用した一行完結の記述を目指しています。このスタイルでは一時オブジェクトが多数生成されるため、**所有権**と**メモリ解放のタイミング**を明示的に扱わないとすぐにクラッシュします。

## NodeDefinitionBaseのクリーンアップフック

- すべての`NodeDefinitionBase`派生クラスは、破棄時に任意のコールバックを呼べるよう`CleanupHook`を保持します。
- `setCleanupHook(hook, context)`で登録し、`clearCleanupHook()`で解除できます。
- デフォルトではフックは設定されず、コピーや代入でもフックは伝播しません（クローンを保持する側が改めて設定する）。
- デストラクタで必ずフックを呼ぶので、保持側は「誰が解放したか分からない」状態を検知可能です。

## NodeCompositionのアリーナ管理

- DSL内で`c.group(...)`や`c.declare(...)`に渡した定義はすべてアリーナにクローンを保管します。
- クローン格納時にクリーンアップフックを`NodeComposition::cleanupStoredNode`へ差し替え、破棄検知後はアリーナから該当スロットを`0`へクリアします。
- `NodeComposition`自身が破棄される際には、保持しているノードのフックを解除し、通常通り`delete`します。
- これにより、DSL内で作られた一時的な`Box()`などが親ノードに所有権を移した後に二重解放される問題を防ぎます。

## DSL（`c.group`）利用のポイント

- `c.group(definition)`は渡した定義のクローンを返し、その所有権は`NodeComposition`のアリーナが持ちます。
- 返り値は`NodeDefinitionBase*`（または具体型のポインタ）なので、`Box() << c.group(...)`のように**所有権を受け取る側が`operator<<`で消費する**設計になります。
- `INestableDefinition::operator<<(NodeDefinitionBase*)`は所有権移転用に実装されているため、DSLマクロを書かなくても自然に扱えます。

## まとめ

1. すべてのノード定義はクリーンアップフックを持ち、寿命を監視できる。
2. `NodeComposition`はアリーナに登録したクローンの寿命を追跡し、二重解放を防ぐ。
3. DSLから返るポインタは所有権を伴うため、`operator<<`に渡すことで安全に木へ取り込める。
4. Nestableな`Fragment`（`F`）を用意しておくと、`F() << child1 << child2`を`c.group(...)`経由で安全に所有権移転でき、`Box`を擬似的なグループ用途にする必要がない。

この設計を文書化しておくことで、今後DSLを拡張する際にも所有権の取り扱いを見失わずに済みます。
