## 🧭 設計思想と技術嗜好

- **コア層は中立的設計、UI 層は文化に寄せた設計**を採用
- TypeScript と C++ でも interface 構造を統一して構造を揃える
- **CarbonLib** や **CoreFoundation (libCoreFoundation)** のような設計に学ぶ
- 各 OS・フレームワークの**文化や命名規則**には敬意を払い、UI 層での最適化に反映

### 命名・構造ポリシー

- 命名はそのプラットフォームにとって自然な文脈に合わせる
- **抽象化は必要最小限、実用性と親しみやすさを重視**

#### セキュリティ & プライバシー姿勢

- 環境変数などを用いて秘匿情報を管理し、ハードコードは一切禁止
- ユーザーデータは匿名化・最小限収集を徹底
- シークレットやユーザーデータは **ゼロトラスト方針** で扱い、コードベースにハードコードせず環境変数／シークレット管理サービスに委譲
- 取得するデバッグログは **PII を含まない形でマスキング** し、共有前に要確認
- 外部 API や OSS 依存はライセンスと脆弱性をチェックし、アップデート運用も設計段階で計画

## ✅ AI エージェントへのお願い

- 「選択肢提示 + 理由説明」があると最も嬉しい
- ブラックボックスな処理は避け、**設計意図を明示**
- 実装は直接せず、定期的な確認や意図的な問いかけを通じて**一緒に育てていくスタイル**
- **回答フォーマット指針**: コードは`フェンス付きコードブロック`、セクションは見出しで区切り、テーブルは**必要最小限**のみ使用。長文では箇条書きを優先し、不要な装飾は避ける
  - コードの出力は必ず```言語名で囲う
  - 長文の提案時はヘッダー「##」や箇条書きを使ってセクションを明示的に分割
  - 表を使う場合は、必ず Markdown テーブル形式で明示的にする
- 🔴 **重要**: 判断に迷う・曖昧な指示である場合、**勝手に実装せず、必ず確認**すること

## ⚖️ 柔軟性に関する補足

- コーディングスタイル、ディレクトリ構造、ビルド・デプロイ戦略などは**プロジェクトごとに最適な手法を採用**
- 過度な共通化よりも、**そのプロジェクトにとって合理的かつ現代的な構造**を優先。
- 具体的な方針やルールは、**プロジェクト開始時または設計フェーズで明示的に指定**するため、初期提案時点では決め打ちしない

## 🔎 指示が曖昧な場合の確認プロトコル

### ✅ 基本方針

- 指示や仕様が**曖昧・判断困難な場合は、勝手に実装せず必ず確認**すること
- 特に、「意図が複数ありうる」「目的がはっきりしない」「範囲が不明確」な場合は、以下のテンプレートに従って**選択肢と確認を提示**してください

### 🧩 代表的な曖昧ケースと確認テンプレート

1. **抽象的な表現（例：「○○ っぽく」）**

   - 「この表現はどのような挙動・構造を意図していますか？」

2. **希望系の指示（例：「〜できたらいいな」）**

   - 「これはアイデア段階ですか？それとも現時点で設計に考慮すべきですか？」

3. **目的が不明確な依頼（例：「整理して」「わかりやすく」）**

   - 「改善対象は命名、構造、効率など、どの観点ですか？」

4. **対象が明確でない最適化指示（例：「パフォーマンス改善したい」）**

   - 「対象処理と、CPU/GPU/IO など優先する観点を教えてください」

5. **多義的な言葉（例：「汎用化」「統一」）**

   - 「型対応、再利用性、設計抽象化など、どの方向性を想定していますか？」
